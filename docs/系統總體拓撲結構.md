# 系統總體拓撲結構

**文檔目標：** 定義 QuantSaaS 全域有哪些物理端、有哪些邏輯模組、狀態如何在它們之間流轉，以及系統整體的生命週期動作。不含任何具體策略公式。

---

## 第 0 章：架構哲學

本系統採用「**狀態最終一致性**」設計。我們不信任網路的穩定性，因此所有行為都以資料庫中的「**目標持倉**」為準。Agent 不斷比對「實際持倉」與「目標持倉」，並自動進行收斂（補單或減倉）。

網路斷線、API 超時、Agent 崩潰都被視為正常事件，而非異常。系統的容錯設計保證：只要 Agent 重新上線，它就能從 SaaS 讀取最新的目標持倉，自動計算差額並補足，無需人工介入。

這一哲學有三個重要推論：

1. **SaaS 是唯一的決策中樞**：`Step()` 只在 SaaS 側執行，Agent 是純粹的「手」，不含任何策略智能。
2. **Agent 的斷線不影響決策連貫性**：SaaS 在 Agent 離線期間跳過下發，等 Agent 回來後自動補齊差額。
3. **沒有「欠債」狀態**：每次 `Step()` 都從當前真實狀態重新決策，不依賴上一次的記憶體變數。

## 第 1 章：三端物理部署形態

### 1.1 SaaS 管理端（雲端，`app_role: saas`）

對策略範本、實例、用戶認證與訂閱進行全量管理。通過 cron 掃描 RUNNING 實例，並在最新已完成的聚合桶尚未處理時執行一次 `Step()`。通過 WebSocket Hub 下發交易指令給已連接的 LocalAgent，並接收成交上報。

**禁區：**
- 嚴禁存放任何形式的交易所 API Secret（連加密存儲都不允許）
- 不得主動向交易所發起下單請求
- 不得執行 GA 進化與回測運算（由 `app_role` 在路由層攔截）

### 1.2 LocalAgent 執行端（用戶本地）

極簡二進位進程，部署於用戶最信任的環境（如家中電腦或加密 VPS）。設定檔（`config.agent.yaml`）內保存 SaaS 連接資訊與交易所 API 憑證。

職責：
1. 啟動時通過 SaaS REST 登錄獲取 JWT
2. 用 JWT 建立 WebSocket 長連接到 SaaS `/ws/agent`
3. 收到 `TradeCommand` 後調用交易所 REST API 執行現貨下單
4. 獲取成交明細與當前餘額，構建 `DeltaReport` 上報 SaaS
5. 定時發送心跳；斷線後以指數退避自動重連

**禁區：** 不得修改策略數學邏輯；不連接資料庫；不自主決策。

### 1.3 進化算力面（本地實驗室，`app_role: lab`）

也是完整的 SaaS 系統，但部署在本地算力機器上。連接與雲端相同的 Postgres 實例，專注執行 GA 進化任務與回測運算。職責是瘋狂計算不同參數下的回報率，產出最優「染色體」。

**禁區：** 不執行真實交易指令下發（`app_role: lab` 在路由層禁用實例啟停與交易下發）。

## 第 2 章：`app_role` 三態行為矩陣

| 能力範圍 | SaaS (Admin) | Agent (Executor) | Lab (Scientist) |
|---------|-------------|-----------------|----------------|
| 策略參數修改 | 唯讀（由 Lab 寫入） | 唯讀 | 寫入（實驗數據） |
| 交易指令發起 | 下發 TradeCommand | 執行 Order | 禁止 |
| 私鑰持有 | 否 | 是 | 否 |
| 歷史數據下載 | 是 | 否 | 是 |
| GA 進化任務 | 禁止（路由攔截） | 禁止 | 完全開放 |
| 實例管理 | 完全開放 | 不涉及 | 唯讀 |

## 第 3 章：邏輯模組與責任邊界

### 3.1 Strategy 策略模組（產品中心）

管理策略範本（圖紙）。定義策略的參數結構、版本標識、交易對類型（現貨）。當前主線策略為 `sigmoid_dca`，面向 BTC/USDT 現貨。

`Step()` 是策略決策的唯一外部入口，所有買賣意圖、倉位規模、風控判斷均在 `Step()` 內產生。回測路徑與實盤路徑共用同一套 `Step()` 實現，不允許內部分叉。

### 3.2 Live/Instance 交易執行模組（設備中心）

實例（Instance）的本質是：策略範本 + 交易標的 + 資金配額 + 用戶授權。實例的狀態（`RUNNING` / `STOPPED` / `ERROR`）全部由此模組管理。

Cron 按基礎掃描頻率檢查所有 `RUNNING` 實例；Tick 只有在最新已完成的聚合桶尚未處理時才推進：拉取 K 線 → 讀取 `PortfolioState` → 調用 `Step()` → 翻譯為 `TradeCommand` → 通過 WebSocket Hub 下發。

### 3.3 Evolution & Backtest 模組（參數實驗室）

通過 GA 遺傳演算法對策略參數進行進化迭代，產出 **挑戰者（challenger）** 參數包。人工在前端審批後可晉升為 **冠軍（champion）**。冠軍參數包用於實例創建或參數更新。只在 `app_role: lab` 或 `dev` 下開放進化任務創建與執行；回測路徑調用與實盤相同的策略 `Step()` 實現。

### 3.4 User/Auth 模組

處理用戶註冊、登錄、JWT 簽發，以及訂閱計畫校驗（創建/啟動實例時做配額守門）。不涉及策略邏輯，不在本文件細化。

## 第 4 章：全域狀態匯流排

### 4.1 單一 Postgres：所有真理的儲存庫

整個系統使用單一 Postgres 資料庫（`quantsaas`），不分庫。存放系統全部的「真相（Truth）」：用戶資產快照、策略染色體、交易歷史、實例狀態、GA 任務與基因記錄。

### 4.2 Redis：僅作緩存，不傳遞信號

Redis 僅作緩存使用（冠軍基因 key、會話 token 等），**不承擔信號傳遞職責**。

> **明確禁止：** 使用 Redis Pub/Sub 傳遞任何交易信號或指令。所有 SaaS → Agent 的指令下發通過 WebSocket 完成。

### 4.3 數據所有權

| 資料 | 真源 | 說明 |
|------|------|------|
| `Portfolio_Actual`（真實持倉） | Agent 上報後持久化 | Agent 是唯一知道交易所真實餘額的節點 |
| `Strategy_Parameters`（染色體） | Lab 產出、SaaS 核准 | 進化引擎寫入 challenger，人工 Promote 為 champion |
| `Trade_Logs`（交易記錄） | Agent 產生、SaaS 儲存 | Agent 上報成交明細，SaaS 持久化並做審計 |
| 交易所 API 憑證 | LocalAgent 本地設定檔 | 唯一合法存放位置，永不進入 SaaS DB |
| GA 任務與基因記錄 | SaaS DB | 單一資料庫，無分庫 |

## 第 5 章：WebSocket 通信協定

### 5.1 連接方向與鑒權

**方向：** Agent（發起）→ SaaS（監聽 `/ws/agent`）。連接建立後 Agent 立即發送含 JWT 的鑒權消息，SaaS 驗證通過後進入正常收發迴圈。

**心跳：** Agent 每 30 秒發送一次 `heartbeat`，SaaS 回 `heartbeat_ack`。

### 5.2 消息類型全表

| 消息類型 | 方向 | 觸發時機 |
|---------|------|---------|
| `auth` | Agent → SaaS | 連接建立後立即，攜帶 JWT |
| `auth_result` | SaaS → Agent | 鑒權完成後返回 |
| `heartbeat` | Agent → SaaS | 每 30 秒 |
| `heartbeat_ack` | SaaS → Agent | 收到心跳後 |
| `command` | SaaS → Agent | `Step()` 產出意圖後，SaaS 翻譯為具體指令下發 |
| `command_ack` | Agent → SaaS | 收到 command 後立即確認（不等執行完成） |
| `delta_report` | Agent → SaaS | 交易執行完成後；或重連時的初始餘額快照 |
| `report_ack` | SaaS → Agent | SaaS 成功處理 DeltaReport 後 |

### 5.3 TradeCommand 語義欄位

| 欄位 | 取值範圍 | 說明 |
|------|---------|------|
| `client_order_id` | 全域唯一字串 | 格式：`inst{id}-{type}-{ts}`，用於去重與審計 |
| `action` | `BUY` / `SELL` | 交易方向 |
| `engine` | `MACRO` / `MICRO` | 產生此指令的引擎層 |
| `symbol` | 如 `BTC/USDT` | 交易對 |
| `amount_usdt` | 字串金額 | 買入時使用（花費 USDT 數量） |
| `qty_asset` | 字串數量 | 賣出時使用（賣出標的資產數量） |
| `lot_type` | `DEAD_STACK` / `FLOATING` | 倉位語義：宏觀底倉 vs 微觀浮倉 |

### 5.4 自癒機制

若連線斷開，Agent 以指數退避重連（初始 1 秒，每次翻倍，最大 5 分鐘）。重連後立即發送初始 `DeltaReport`（當前交易所餘額快照，不含 `client_order_id` 與 `execution`）。

如果發現帳戶裡的 BTC 數量與資料庫要求的不符，SaaS 在下次 Tick 基於真實餘額重新決策，自動計算差額並補足，不需要人工干預。

## 第 6 章：系統級生命週期動作

### 6.1 系統初始化

**觸發：** SaaS 進程啟動時。

1. 讀取 `config.yaml`，初始化 DB 連接、Redis 連接、JWT 秘鑰
2. 執行 GORM AutoMigrate，以 Go struct 為唯一 schema 真源完成結構同步
3. 從 DB 讀取所有狀態為 `RUNNING` 的實例，載入記憶體管理清單
4. 啟動 WebSocket Hub，開始接受 Agent 連接
5. 啟動 cron 調度器，註冊實例基礎掃描任務

### 6.2 Cron Tick 驅動實例 Step()

**觸發：** cron 命中 `RUNNING` 實例，且最新已完成的聚合桶尚未處理。

1. 從交易所公開 API 拉取 K 線歷史資料
2. 從 DB 讀取 `PortfolioState`（由最新 DeltaReport 維護的真實帳戶狀態）
3. 從 DB 讀取 `RuntimeState`（策略內部運行狀態快照）
4. 構建 `StrategyInput`（含收盤價序列、時間戳記序列、資產狀態）
5. 調用 `Step()`（**只在 SaaS 側執行**，Agent 不含此邏輯）
6. 翻譯為 `TradeCommand`（含全域唯一 `client_order_id`）
7. 在 DB 寫入 pending 執行記錄
8. 通過 WebSocket Hub 下發給對應 Agent

若 Agent 當前未連接，記錄警告並跳過下發，等待下次 Tick 重試。

### 6.3 Agent 斷線重連

1. 指數退避等待後，重新登錄獲取新 JWT
2. 建立新 WebSocket 連接，發送 `auth`
3. 連接建立後立即發送初始 `DeltaReport`（餘額快照，不含執行記錄）
4. SaaS 收到快照後更新相關實例的 `PortfolioState`
5. 下次 cron Tick 時，SaaS 基於真實餘額自動糾偏，無需人工干預

### 6.4 優雅停機（Graceful Shutdown）

**觸發：** SaaS 進程接收到 `SIGTERM`。

1. 停止接受新的 Agent 連接與 cron Tick 任務
2. 等待正在執行中的 Tick 完成（超時保護）
3. 持久化所有活躍實例的 `RuntimeState` 快照，避免重啟後策略狀態撕裂
4. 關閉所有 WebSocket 連接，發送關閉幀
5. 關閉 DB 與 Redis 連接

## 第 7 章：不可推翻的技術決策

違反任何一條時必須立即停止開發並報告，不得繞過。

**1. 策略同構**
回測與實盤必須調用同一個 `Step()` 實現。`Step()` 內部禁止出現 `if isBacktest` 類分支。`Step()` 只在 SaaS 側執行，Agent 二進位不包含任何策略代碼。

**2. 策略純函數**
策略代碼內部絕對禁止：計時器、網路請求、資料庫讀寫、任何文件 I/O。`Step()` 是純函數，只依賴傳入的 `StrategyInput`，只輸出 `StrategyOutput`。

**3. API Key 物理隔離**
交易所 API Key / Secret 只允許存在於 LocalAgent 的本地設定檔（`config.agent.yaml`）中，永不進入 SaaS，永不寫入資料庫，永不通過網路傳輸。

**4. GORM AutoMigrate 為唯一 schema 真源**
資料庫結構以 Go struct 為真源，通過 GORM AutoMigrate 管理。不寫 SQL migration 文件，不維護版本化 migration 腳本。

**5. 無量綱計算**
所有價格相關計算使用對數收益率或比率，禁止用絕對價格做跨標的比較。

**6. 倉位三態語義**
資產狀態以 `DeadStack` / `FloatStack` / `ColdSealedStack` 三態區分語義，不能混用。三態更新規則由策略 `Step()` 決定，SaaS 按 `lot_type` 字段落賬。

**7. 單一資料庫，Redis 不傳信號**
SaaS 使用單一 Postgres 實例，不分庫；Redis 僅作緩存，不作信號傳遞通道。
