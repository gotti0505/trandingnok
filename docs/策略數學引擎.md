# 策略數學引擎

**文檔目標：** 定義 `Step()` 函數契約、市場狀態感知層邏輯、宏觀/微觀引擎算法規格、Sigmoid 公式、倉位三態語義，以及可進化參數（染色體）的邊界定義。本文檔是所有策略代碼實現的唯一依據。

---

## 第 0 章：設計原則

本策略面向 BTC/USDT 現貨市場，核心哲學是：

1. **長期複利優先**：透過宏觀引擎持續累積底倉（DeadBTC），資金規模隨權益正向回饋滾動增長。
2. **天平動態平衡**：透過微觀引擎（Sigmoid 天平）在高位將 FloatBTC 換回 USDT，補充後續 DCA 的資金彈藥。
3. **市場三態保護**：在市場劇烈波動期或安靜期暫停操作，節省手續費，避免粉塵磨損。
4. **策略同構**：回測與實盤共用同一個 `Step()` 實現，內部無任何 `if isBacktest` 分支。

**複利迴路：**

```
SpendableUSDT → (宏觀 DCA 買入) → DeadStack
DeadStack → (ROI 達標釋放) → FloatStack
FloatStack → (微觀 Sigmoid 高點賣出) → SpendableUSDT
```

每一輪迴路，資金規模隨權益正向回饋滾動增長，複利可論證。

## 第 1 章：Step() 契約

### 1.1 函數簽名

```go
func Step(input StrategyInput) StrategyOutput
```

`Step()` 是策略決策的**唯一外部入口**。所有買賣意圖、倉位規模、風控判斷均在此函數內產生。

### 1.2 輸入（StrategyInput）

| 欄位 | 類型 | 說明 |
|------|------|------|
| `Closes` | `[]float64` | 收盤價序列（時間昇序，最早在前） |
| `Timestamps` | `[]int64` | 對應時間戳記序列（毫秒） |
| `Portfolio` | `PortfolioState` | 當前真實資產狀態（由最新 DeltaReport 維護） |
| `Runtime` | `RuntimeState` | 策略內部持久化狀態（EMA 快照、當月最高價、已觸發檔位等） |
| `Params` | `Chromosome` | 當前激活的染色體參數包 |

**PortfolioState 關鍵欄位：**

| 欄位 | 說明 |
|------|------|
| `TotalUSDT` | 帳戶總 USDT 餘額 |
| `ReserveFloor` | 安全底線（出生點固定，永不動用） |
| `SpendableUSDT` | 可用資金 = `TotalUSDT - ReserveFloor` |
| `DeadStack` | 底倉 BTC 數量（宏觀引擎買入，只進不出） |
| `FloatStack` | 浮倉 BTC 數量（微觀引擎可賣出） |
| `ColdSealedStack` | 冷封存 BTC（永不動用） |
| `NAVInitial` | 初始淨資產，用於計算 ROI |

### 1.3 輸出（StrategyOutput）

| 欄位 | 類型 | 說明 |
|------|------|------|
| `Intents` | `[]TradeIntent` | 本次 Tick 產生的交易意圖列表（可為空切片） |
| `Runtime` | `RuntimeState` | 更新後的策略內部狀態，SaaS **必須**持久化 |

**TradeIntent 欄位：**

| 欄位 | 說明 |
|------|------|
| `Action` | `BUY` / `SELL` |
| `Engine` | `MACRO` / `MICRO` |
| `LotType` | `DEAD_STACK` / `FLOATING` |
| `AmountUSDT` | 買入時填寫（USDT 數量） |
| `QtyAsset` | 賣出時填寫（BTC 數量） |

### 1.4 純函數鐵律

`Step()` **禁止**執行以下任何操作：
- 計時器或 goroutine 啟動
- 網路請求（HTTP / WebSocket / gRPC）
- 資料庫讀寫（任何形式）
- 文件 I/O
- 不可復現的隨機數（不由外部傳入 seed）

`Step()` 的輸出只由輸入確定。相同的 `StrategyInput` **必然**產生相同的 `StrategyOutput`。

## 第 2 章：市場狀態感知層

我們將市場簡化為三種狀態，避免在不適當的市場環境下過度交易。

### 2.1 狀態判斷邏輯

| 市場狀態 | 判斷條件 | 後續動作 |
|---------|---------|---------|
| **安靜（Quiet）** | 24 小時波動率 `σ₂₄ₕ < 1.5%` | **關閉所有交易**，節省手續費（避免粉塵交易磨損） |
| **熊市（Bear）** | 當前收盤價 < EMA₂₀₀ | 激活宏觀引擎（DCA 買入），暫停微觀獲利了結 |
| **牛市（Bull）** | 當前收盤價 > EMA₂₀₀ 且 EMA₂₀₀ 斜率向上 | 激活微觀引擎（Sigmoid 賣出），宏觀引擎待機 |

**優先級：安靜 > 熊市 > 牛市。** 若 `σ₂₄ₕ < 1.5%`，無論趨勢如何，當次 Tick 不產生任何交易意圖，直接返回空 `Intents`。

### 2.2 EMA₂₀₀ 計算

200 日指數移動平均線，使用標準 EMA 遞推公式：

$$EMA_t = \alpha \cdot P_t + (1 - \alpha) \cdot EMA_{t-1}, \quad \alpha = \frac{2}{201}$$

EMA₂₀₀ 的值持久化在 `RuntimeState` 中，每次 Tick 增量更新，無需重算全序列。

**斜率判斷：** 比較當前 EMA₂₀₀ 與 10 根 K 線前的 EMA₂₀₀ 值：

$$slope = EMA_t - EMA_{t-10}$$

`slope > 0` 則判定為向上趨勢（牛市條件之一）。

### 2.3 24 小時波動率（σ₂₄ₕ）

取最近 24 根小時 K 線的最高價與最低價：

$$\sigma_{24h} = \frac{High_{24h} - Low_{24h}}{Low_{24h}}$$

若 `σ₂₄ₕ < 0.015`（即 1.5%），判定市場安靜。

## 第 3 章：微觀引擎（Sigmoid 天平）

微觀引擎在**牛市**狀態下激活，作用是：在高位將 FloatBTC 逐步轉換回 USDT，實現波段獲利了結，補充 SpendableUSDT 供未來 DCA 使用。

### 3.1 Sigmoid 公式

以 EMA₂₀₀ 為基準，用對數比率（無量綱）衡量當前價格的「高溢價」程度：

$$x = k \cdot \ln\!\left(\frac{P_{current}}{EMA_{200}}\right)$$

$$s = \frac{1}{1 + e^{-x}}$$

其中 $k$ = `sigmoid_sensitivity`（染色體參數，範圍 1.0 ~ 10.0）。

**天平語義：**

| $s$ 值域 | 含義 | 動作 |
|---------|------|------|
| $s = 0.5$ | 價格剛好在 EMA₂₀₀，天平平衡 | 不操作 |
| $s > 0.5$ | 價格高於 EMA₂₀₀，天平偏向賣出 | 賣出 FloatBTC |
| $s < 0.5$ | 價格低於 EMA₂₀₀ | 微觀引擎不介入（交棒給宏觀） |

**靈敏度效果：** $k$ 越大，同樣的價格偏離對應的賣出信號越強，系統在高位更激進地獲利了結；$k$ 越小，系統需要價格大幅偏離 EMA₂₀₀ 才開始行動。

### 3.2 每 Tick 賣出量計算

只有當 $s > 0.5$ 時才產生賣出意圖：

$$sell\_ratio = (s - 0.5) \times 2 \quad (\in [0, 1])$$

$$sell\_qty = FloatStack \times sell\_ratio \times r_{tick}$$

其中 $r_{tick} = 0.05$（硬編碼常數），表示每個 Tick 最多賣出 FloatStack 的 5%，防止單次清倉。

若 `FloatStack == 0`，微觀引擎直接跳過。

### 3.3 止損邏輯（優先於天平運算）

在微觀引擎的天平運算**之前**，先判斷止損條件：

$$ROI = \frac{NAV_{current} - NAV_{initial}}{NAV_{initial}}$$

$$NAV_{current} = TotalUSDT + (DeadStack + FloatStack + ColdSealedStack) \times P_{current}$$

若 $ROI < stop\_loss\_threshold$（染色體參數，範圍 -30% ~ -10%）：

1. 將所有 `FloatStack` 市價賣出 → USDT（`lot_type = FLOATING`）
2. 在 `RuntimeState` 記錄止損觸發標記
3. 本次 Tick 不再執行其他交易意圖

## 第 4 章：宏觀引擎（DCA 策略）

宏觀引擎在**熊市**狀態下激活（安靜市場除外），作用是：在價格下跌時分批買入 BTC，將 SpendableUSDT 逐步轉化為 DeadBTC，不看技術指標，只看「夠不夠便宜」。

### 4.1 價格百分比階梯

策略設計採用「價格百分比階梯」。以當月（自然月）最高收盤價 `P_monthHigh` 為基準：

| 觸發條件 | 投入比例 |
|---------|---------|
| `P < P_monthHigh × (1 - dca_drop_pct × 1.0)` | `SpendableUSDT × 5%` |
| `P < P_monthHigh × (1 - dca_drop_pct × 1.5)` | `SpendableUSDT × 10%` |
| `P < P_monthHigh × (1 - dca_drop_pct × 2.0)` | `SpendableUSDT × 15%` |

其中 `dca_drop_pct` 為染色體參數（範圍 0.05 ~ 0.30）。

**範例：** `dca_drop_pct = 0.10` 時，三個觸發點為「從本月最高點下跌 10%、15%、20%」，分別投入 SpendableUSDT 的 5%、10%、15%。

### 4.2 月度重置規則

每進入新的自然月，`P_monthHigh` 重置為當月第一根 K 線的收盤價，`RuntimeState` 中清空已觸發檔位記錄。**同一個月內，同一個檔位不重複觸發**（防止震盪市中連續觸發同一層）。

### 4.3 宏觀買入的落賬規則

宏觀引擎買入的 BTC 全部歸入 `DeadStack`（底倉），不進入 `FloatStack`。TradeIntent 的 `LotType = DEAD_STACK`，SaaS 在 DeltaReport 回填時按此欄位落賬。

### 4.4 SpendableUSDT 的定義

$$SpendableUSDT = TotalUSDT - ReserveFloor$$

`ReserveFloor` 為資金安全底線（出生點固定參數，GA 不進化）。宏觀引擎的百分比計算以 `SpendableUSDT` 為分母，而非 `TotalUSDT`，確保 `ReserveFloor` **永遠不被動用**。

## 第 5 章：倉位三態語義

資產的語義分類是整個策略運轉的基礎。三態嚴格分離，不可混用。

| 倉位態 | 英文標識 | 語義 | 操作規則 |
|-------|---------|------|---------|
| **底倉** | `DeadStack` | 宏觀引擎買入的長期 BTC | 只進不出；唯一出口是第 6 章的釋放規則 |
| **浮倉** | `FloatStack` | 可由微觀引擎賣出的 BTC | 由底倉釋放補充；微觀引擎消耗 |
| **冷封存** | `ColdSealedStack` | 人工指定永不動用的 BTC | `Step()` 不可讀取、不可操作此欄位 |

**三態不可逾越：**
- `DeadStack → FloatStack`：只能通過第 6 章定義的 ROI 觸發釋放規則。
- `FloatStack → DeadStack`：逆向操作被明確禁止。
- `ColdSealedStack`：任何引擎均不得觸碰。

## 第 6 章：DeadBTC 釋放規則（獲利了結）

### 6.1 釋放條件

當**整體投資組合收益率（ROI）** 達到釋放閾值時，觸發底倉語義轉換：

$$ROI_{trigger} = 50\%$$（出生點固定參數，GA 不進化）

$$release\_qty = DeadStack \times 20\%$$

將 `release_qty` 數量的 BTC **從 `DeadStack` 移入 `FloatStack`**。

此操作的性質：
- **只改變 SaaS 側帳本的語義分類**，不向 Agent 下發任何 `TradeCommand`
- **必須**寫審計日誌（記錄：釋放數量、觸發時間、觸發 ROI）
- 釋放後 ROI 不重置；下次釋放需 ROI 再次升至 50% 以上

### 6.2 釋放後的流轉鏈條

```
DeadStack  ──(ROI≥50% 釋放)──▶  FloatStack  ──(Sigmoid 高點賣出)──▶  USDT
                                                                         │
                                                                         ▼
                                                              SpendableUSDT（補充 DCA 彈藥）
```

FloatBTC 進入 FloatStack 後，在下次 Tick 的微觀引擎運算中，若價格高於 EMA₂₀₀ 且 Sigmoid 信號足夠強，會逐步透過市價賣單轉換為 USDT。

### 6.3 複利的可論證性

這個設計滿足複利的數學前提：

- **底倉只進不出**：資金規模的下限隨每次 DCA 買入單調遞增。
- **浮倉回收再投**：賣出 FloatBTC 得到的 USDT 補充 SpendableUSDT，供下一輪 DCA 使用，驅動下一次底倉擴大。
- **ReserveFloor 永不動用**：安全底線確保系統在極端市場下不歸零。

## 第 7 章：可進化參數契約（Chromosome）

### 7.1 GA 搜索空間（可進化參數）

這些參數由 Lab 側 GA 進化引擎搜索，每次 Promote 時更新到實例。

| 參數名 | 說明 | 搜索邊界 |
|-------|------|---------|
| `dca_drop_pct` | 觸發 DCA 買入的基礎跌幅（階梯的基準單位） | [0.05, 0.30] |
| `sigmoid_sensitivity` | Sigmoid 天平的靈敏度（$k$ 值） | [1.0, 10.0] |
| `stop_loss_threshold` | 止損觸發的最大虧損率（負數） | [-0.30, -0.10] |

### 7.2 出生點參數（固定，不參與進化）

這些參數在實例創建時固定，GA 不搜索、不變異、不交叉。

| 參數名 | 說明 | 範例值 |
|-------|------|-------|
| `base_currency` | 計價貨幣 | `"USDT"` |
| `trading_pair` | 交易對 | `"BTC/USDT"` |
| `initial_capital` | 初始本金（USDT） | 由用戶設定 |
| `reserve_floor` | 永不動用的安全底線（USDT） | 由用戶設定 |
| `roi_release_trigger` | DeadBTC 釋放的 ROI 閾值 | `0.50`（50%） |
| `release_ratio` | 每次釋放 DeadStack 的比例 | `0.20`（20%） |
| `monthly_inject` | 每月定期注資金額（USDT，可為 0） | 由用戶設定 |

### 7.3 染色體 JSON 編碼格式

GA 染色體以 JSON 格式存入資料庫 `param_pack` 欄位：

```json
{
  "spawn_point": {
    "base_currency": "USDT",
    "trading_pair": "BTC/USDT",
    "initial_capital": 10000,
    "reserve_floor": 1000,
    "roi_release_trigger": 0.50,
    "release_ratio": 0.20,
    "monthly_inject": 500
  },
  "sigmoid_dca_config": {
    "dca_drop_pct": 0.10,
    "sigmoid_sensitivity": 5.0,
    "stop_loss_threshold": -0.20
  }
}
```

### 7.4 邊界修復（Clamp）規則

GA 交叉或變異後，所有可進化參數必須通過 `Clamp` 函數修復到合法邊界：

```
dca_drop_pct         → clamp([0.05, 0.30])
sigmoid_sensitivity  → clamp([1.0, 10.0])
stop_loss_threshold  → clamp([-0.30, -0.10])
```

`stop_loss_threshold` 為負數，`clamp` 時注意方向：`-0.30` 為下界（虧損更多），`-0.10` 為上界（更快止損）。
