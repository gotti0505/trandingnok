**文檔目標:** 定義 QuantSaaS 全域有哪些物理端、有哪些邏輯模組、狀態如何在它們之間流轉，以及系統整體的生命週期動作。不含任何具體策略公式。

---

## 1. 架構哲學：時空正交與無記憶解算

**動作原則：** 所有業務推進必須被封裝為離散的"動作"，動作之間嚴禁隱式狀態傳遞（禁止"欠債記錄"）。每個動作讀入當前狀態快照，產出新的確定性意圖，不依賴上一動作的記憶體變數。

**窗口原則：** 每個模組只能讀取屬於自己的"資料視窗"和"資金視窗"。策略模組看不到資料庫內部狀態，宏觀引擎看不到微觀的浮倉細節，進化引擎看不到實盤帳戶。

## 2. 三端物理部署形態

### 2.1 SaaS 管理端（雲端，`app_role: saas`）

對策略範本、實例、使用者認證與訂閱進行全量管理。通過 cron 掃描 RUNNING 實例，並僅在該實例最新已完成的 `[聚合週期]` 聚合桶尚未處理時執行一次 `Step()`。通過 WebSocket Hub 下發交易指令給連接中的 LocalAgent，並接收成交上報。

**不執行：** GA 進化與回測運算（可通過 `app_role` 在路由層攔截寫介面）。

**不保存：** 任何交易所 API 憑證。

### 2.2 LocalAgent 執行端（用戶本地）

極簡二進位進程，使用者本地運行。設定檔（`config.agent.yaml`）內保存 SaaS 連接資訊與 [你的交易所] API 憑證。

職責：
1. 啟動時通過 SaaS REST 登錄獲取 JWT
2. 用 JWT 建立 WebSocket 長連接到 SaaS `/ws/agent`
3. 收到 `TradeCommand` 後調用 [你的交易所] REST API 執行現貨下單
4. 獲取成交明細與當前餘額，構建 `DeltaReport` 上報 SaaS
5. 定時發送心跳；斷線後以指數退避自動重連

**不含：** 任何策略代碼，不連接資料庫，不自主決策。

### 2.3 進化算力面（本地實驗室，`app_role: lab`）

也是完整 SaaS 系統，但部署在本地算力機器上。資料庫連接雲端 SaaS 的同一 Postgres 實例，專注執行 GA 進化任務與回測運算。

**不執行：** 真實交易指令下發（`app_role: lab` 在路由層禁用實例啟停與交易下發）。

### 2.4 `app_role` 三態行為矩陣

| 角色 | 部署場景 | 開放能力 | 限制 |
|------|---------|---------|------|
| `saas` | 雲端 SaaS | 實例管理、交易指令下發、使用者運營 | 禁止進化/回測寫介面 |
| `lab` | 本地實驗室 | GA 進化、回測、基因庫管理 | 禁止交易指令下發、實例唯讀 |
| `dev` | 開發測試 | 所有模組全開 | 無限制 |

## 3. 邏輯模組與責任邊界

### 3.1 Strategy 策略模組（產品中心）

管理策略範本（圖紙）。定義策略的參數結構、版本標識、交易對類型（現貨/合約）。當前主線策略為 `[your-strategy-id]`，面向 [你的交易所] 現貨 [你的交易對]。

`Step()` 是策略決策的唯一外部入口，所有買賣意圖、倉位元規模、風控判斷均在 `Step()` 內產生。回測路徑與實盤路徑共用同一套 `Step()` 實現，不允許內部分叉。

### 3.2 Live/Instance 交易執行模組（設備中心）

實例（Instance）的本質是：策略範本 + 交易標的 + 資金配額 + 用戶授權。實例的創建、啟動、停止、刪除，以及運行中的狀態（`RUNNING` / `STOPPED` / `ERROR`）全部由此模組管理。

cron 按基礎掃描頻率檢查所有 `RUNNING` 狀態實例；實例 tick 只有在最新已完成的 `[聚合週期]` 聚合桶未處理過時才推進：拉取公開 K 線數據，讀取 `PortfolioState`，構建 `StrategyInput`，調用 `Step()`，將產出的交易意圖翻譯為 `TradeCommand`，通過 WebSocket Hub 下發給已連接的對應 Agent。

### 3.3 Evolution & Backtest 模組（參數實驗室）

通過 GA 遺傳演算法對策略參數進行進化反覆運算，產出 **挑戰者（challenger）** 參數包。人工在前端審批後可將挑戰者晉升為 **冠軍（champion）**。冠軍參數包用於實例創建或參數更新。

只在 `app_role: lab` 或 `dev` 下開放進化任務創建與執行，回測路徑調用與實盤相同的策略 `Step()` 實現。

### 3.4 User/Auth 模組

處理用戶註冊、登錄、JWT 簽發，以及訂閱計畫校驗（創建實例、啟動實例時做配額守門）。此模組不涉及策略邏輯，不在本文檔細化。

## 4. 全域狀態匯流排

### 4.1 單一 Postgres + 單一 Redis

整個系統使用單一 Postgres 資料庫（`quantsaas`），不再區分 Shell DB 與 Live DB。Redis 僅作緩存使用（冠軍基因 key、會話等），不承擔信號傳遞職責。

### 4.2 數據真源與所有權

| 資料 | 真源 | 說明 |
|------|------|------|
| 策略決策邏輯 | `Step()` 實現（SaaS 側執行） | 回測與實盤必須共用 |
| 冠軍參數包 / 範本 | SaaS DB | 進化產出，供實例綁定 |
| 終端使用者與訂閱 | SaaS DB | 統一在 SaaS 管理 |
| 運行中實例狀態 | SaaS DB | 由 Agent 上報後維護 |
| 交易所 API 憑證 | LocalAgent 本地設定檔 | 僅 Agent 持有，永不進入 SaaS DB |
| 成交記錄與資產快照 | SaaS DB | Agent 上報後持久化 |
| GA 任務與基因記錄 | SaaS DB | 單一資料庫，無分庫 |

### 4.3 交易所憑證物理隔離鐵律

[你的交易所] API Key / Secret / Passphrase **只能**存在於 LocalAgent 的本地設定檔（`config.agent.yaml`）中。

- 永不進入 SaaS 服務
- 永不寫入任何資料庫表
- 永不通過網路傳輸到雲端

發現代碼將 API Key 寫入 SaaS 側時，必須立即停止並報告。

## 5. IoT 下發與上報通信協議

雲端與端側嚴格遵循"**雲端只信上報，端側無腦執行**"機制，徹底解決網路斷線、丟包與髒資料問題。

### 5.1 WebSocket 長連接模型

Agent 發起連接，SaaS 監聽 `/ws/agent` 端點。連接建立時 Agent 必須先發送含 JWT 的鑒權消息，SaaS 驗證通過後返回鑒權結果，之後進入正常收發迴圈。Agent 每 30 秒發送一次心跳，SaaS 回確認。

### 5.2 消息類型全表

| 消息類型 | 方向 | 觸發時機 |
|---------|------|---------|
| `auth` | Agent → SaaS | 連接建立後立即發送，攜帶 JWT |
| `auth_result` | SaaS → Agent | 鑒權完成後返回 |
| `heartbeat` | Agent → SaaS | 每 30 秒 |
| `heartbeat_ack` | SaaS → Agent | 收到心跳後 |
| `command` | SaaS → Agent | `Step()` 產出意圖後，SaaS 翻譯為具體指令下發 |
| `command_ack` | Agent → SaaS | 收到 command 後立即確認（不等執行完成） |
| `delta_report` | Agent → SaaS | 交易執行完成後；或重連時的初始餘額快照 |
| `report_ack` | SaaS → Agent | SaaS 成功處理 DeltaReport 後 |

### 5.3 TradeCommand 語義

每條 `TradeCommand` 攜帶以下語義欄位：

| 欄位 | 取值範圍 | 說明 |
|------|---------|------|
| `client_order_id` | 全域唯一字串 | 格式：`inst{id}-{type}-{ts}`，用於去重與審計 |
| `action` | `BUY` / `SELL`（大寫） | 交易方向 |
| `engine` | `MACRO` / `MICRO`（大寫） | 產生此指令的引擎層 |
| `symbol` | 如 `[你的交易對]` | 交易對 |
| `amount_usdt` | 字串金額 | 買入時使用，表示花費 USDT 數量 |
| `qty_asset` | 字串數量 | 賣出時使用，表示賣出標的資產數量 |
| `lot_type` | `DEAD_STACK` / `FLOATING` | 倉位語義，宏觀吸入底倉 vs 微觀浮動倉 |

SaaS 下發 TradeCommand 的同時在資料庫寫一條 pending 記錄，供 DeltaReport 回填時按 `LotType` 拆分更新 `DeadStack` / `FloatStack`。

### 5.4 DeltaReport 語義

Agent 執行完畢後上報，包含：
- `balances`：當前所有資產的餘額快照（標的資產與 USDT 的可用與凍結）
- `execution`：本次成交明細（成交量、成交價格、手續費、狀態）
- `client_order_id`：對應的指令 ID（重連時發送初始快照時可為空）

### 5.5 狀態收斂與天然自愈

SaaS 收到 DeltaReport 後刷新對應實例的 `PortfolioState`，持久化 `SpotExecution` 與 `TradeRecord`，並寫審計日誌。

下次 `Step()` 調用時，系統基於真實餘額與語義拆分重新決策。這意味著：
- Agent 斷線後 SaaS 跳過該實例的 tick，無須人工干預
- Agent 重連後立即發送初始餘額快照，SaaS 在下次 tick 自動糾偏
- SaaS 重啟後從 DB 恢復 RUNNING 實例清單，等待 Agent 重連

## 6. 系統級生命週期動作

### 6.1 系統初始化（System Init）

**觸發時機：** SaaS 進程啟動時。

**動作流轉：**
1. 讀取設定檔（`config.yaml`），初始化 DB 連接、Redis 連接、JWT 秘鑰
2. 執行 GORM AutoMigrate，以 Go struct 為唯一 schema 真源完成資料庫結構同步
3. 從 DB 讀取所有狀態為 `RUNNING` 的實例，註冊到記憶體管理清單
4. 啟動 WebSocket Hub，開始接受 Agent 連接
5. 啟動 cron 調度器，註冊實例基礎掃描任務；具體 `Step()` 推進由每個實例的聚合桶去重控制

### 6.2 Cron Tick 驅動實例 Step()

**觸發時機：** cron 基礎掃描命中 `RUNNING` 實例後，若該實例最新已完成的聚合桶尚未處理，則觸發一次 `Step()`；已處理過同一桶時跳過，避免實盤比回測多跑決策。

**動作流轉：**
1. 從交易所公開 API 拉取 K 線歷史資料
2. 從 DB 讀取實例的 `PortfolioState`（由上次 DeltaReport 維護的真實帳戶狀態）
3. 從 DB 讀取 `RuntimeState`（策略內部運行狀態快照）
4. 構建 `StrategyInput`（含 K 線序列、資產狀態、時間戳記）
5. 調用 `panel.Step()`（**只在 SaaS 側執行**，Agent 不含此邏輯）
6. 將 `Step()` 產出的交易意圖翻譯為 `TradeCommand`（含全域唯一 `client_order_id`）
7. 在 DB 寫入 pending 執行記錄
8. 通過 WebSocket Hub 將 `TradeCommand` 下發給對應 Agent

若 Agent 當前未連接，則跳過下發，記錄警告，等待下次 tick 時重試。

### 6.3 Agent 斷線重連

**觸發時機：** Agent 檢測到 WebSocket 連接斷開。

**動作流轉：**
1. 等待初始退避時間（1 秒），嘗試重新登錄獲取 JWT
2. 建立新的 WebSocket 連接
3. 連接建立後立即發送初始 `DeltaReport`（當前交易所餘額快照，不含 `client_order_id` 與 `execution`）
4. SaaS 收到快照後更新相關實例的 `PortfolioState`
5. 下次 cron tick 時，SaaS 基於真實餘額重新決策，無需人工干預

重連退避策略：初始 1 秒，每次翻倍，最大等待 5 分鐘。

### 6.4 優雅停機與狀態快照（Graceful Shutdown）

**觸發時機：** SaaS 進程接收到作業系統中斷信號（如 `SIGTERM`）。

**動作流轉：**
1. 停止接受新的 Agent 連接與 cron tick 任務
2. 等待正在執行中的 tick 完成（超時保護）
3. 持久化所有活躍實例的 `RuntimeState` 快照，避免重啟後策略狀態撕裂
4. 關閉所有 WebSocket 連接，發送關閉幀
5. 關閉 DB 與 Redis 連接

## 7. 不可推翻的技術決策

以下為整個系統的硬約束，違反時必須停止開發並報告，不得繞過。

**1. 複利前置條件（鐵律）**
任何涉及策略機制的設計或實現，必須能清楚說明複利如何發生（資金規模隨權益正回饋滾動）。不滿足可論證複利性質的策略設計，不進行開發。

**2. 策略同構原則**
回測與實盤必須調用同一個 `Step()` 實現。`Step()` 內部禁止出現 `if isBacktest { ... } else { ... }` 類分支。`Step()` 只在 SaaS 側執行，Agent 二進位不包含任何策略代碼。

**3. 策略內部純淨原則**
策略代碼內部絕對禁止：計時器、網路請求（HTTP / WebSocket / gRPC）、資料庫讀寫、任何檔 I/O。策略是純函數，只依賴傳入的 `StrategyInput`，只輸出 `StrategyOutput`。

**4. 現貨策略 ACL OHLC 剝離**
`Manifest.IsSpot == true` 的策略，內核代碼禁止依賴 `OHLCV` 聚合結構（`quant.Bar`）。所有 OHLCV 到 `[]float64 + []int64`（收盤價序列 + 時間戳記序列）的降級必須在 ACL 外圈（回測適配器入口與實盤 tick 入口）完成。

**5. API Key 物理隔離**
[你的交易所] API Key / Secret / Passphrase 只允許存在於 LocalAgent 的本地設定檔中，永不進入 SaaS，永不寫入資料庫，永不通過網路傳輸。

**6. GORM AutoMigrate 為唯一 schema 真源**
資料庫結構以 Go struct 為真源，通過 GORM AutoMigrate 管理。不寫 SQL migration 檔，不維護版本化 migration 腳本。

**7. 無量綱計算**
所有價格相關計算使用對數收益率或比率（無量綱），禁止用絕對價格做跨標的比較。

**8. 倉位三態**
資產狀態以 `DeadStack`（宏觀長期底倉）/ `FloatStack`（微觀浮動倉）/ `ColdSealedStack`（冷封存，永不釋放）三態區分語義，不能混用。倉位元三態的更新規則由策略 `Step()` 決定，SaaS 按 `LotType` 字段落賬。

**9. 底倉釋放僅在 SaaS 側更新，不下發 Agent**
`dead_stack → floating` 的語義轉換只改變 SaaS 側的 `PortfolioState` 帳本，不向 Agent 下發任何 `TradeCommand`，但必須寫審計日誌。

**10. 單一資料庫，無分庫**
SaaS 使用單一 Postgres 實例，不分 Shell DB 與 Live DB，不使用 Redis 作為信號傳遞通道（Redis 僅作緩存）。

**11. 使用者介面零數學**
面向使用者的字串中避免出現內部狀態機術語、無上下文的裸數學量名或公式片段。希臘字母並非一律禁止；若含義上必須保留，應採用「字母 + 文字釋義」並置（例如「σ（標準差）」「ρ（相關係數）」），禁止僅展示希臘字母而無對應說明。
